# PepeCore v2

بازنویسی کامل پروژه با یک هدف مشخص: **جدا کردن هسته وایرگارد از پنل.**

---

## مشکل نسخه قبلی

در نسخه قدیم، پرایوت کی سرف‌شارک بخشی از فرایند inject بود. مسیرش این بود:

```
تغییر پرایوت کی
  → دوباره inject بزن
  → اینباندها از اول ساخته می‌شن
  → UUID ها عوض می‌شن
  → پورت‌ها جابه‌جا می‌شن
  → کانفیگ کاربرها می‌سوزه
```

ریشه‌ی مشکل این بود که کلید، سرور و کشور همه داخل تگ‌های پنل کدگذاری شده بودند
(`B-Out-Japan-XX1`). پس هر تغییری در آن‌ها یعنی تگ عوض می‌شود، و تگ که عوض شود
یعنی کل ساختار پنل باید دوباره ساخته شود.

---

## راه‌حل: مفهوم Slot

اسلات یک **پورت SOCKS5 ثابت و دائمی** روی `127.0.0.1` است:

```
slot 0  →  127.0.0.1:20000   ← این هرگز عوض نمی‌شود
slot 1  →  127.0.0.1:20001   ← این هرگز عوض نمی‌شود
slot 2  →  127.0.0.1:20002   ← این هرگز عوض نمی‌شود
```

پنل فقط به همین **پورت‌ها** بایند می‌شود — نه به کلید، نه به کشور، نه به سرور.

آنچه *پشت* اسلات است (کدام کلید، کدام سرور سرف‌شارک، کدام کشور) هر لحظه قابل
تعویض است. هسته آن را زنده عوض می‌کند و پنل اصلاً خبردار نمی‌شود، چون از دید
Xray پورت `127.0.0.1:20003` هنوز همان `127.0.0.1:20003` است.

### نتیجه

```
تغییر پرایوت کی
  → N تا فایل کانفیگ بازنویسی می‌شود
  → N تا پروسه ری‌استارت می‌شود
  → تمام
```

صفر تماس با API پنل. صفر بازسازی اینباند. صفر تغییر UUID.
کاربرها در تمام مدت عملیات وصل می‌مانند.

## معماری

```
core/registry.py   قرارداد پایدار: نگاشت slot → port، استخر کلیدها
core/engine.py     هسته مستقل وایرگارد. هیچ چیزی از پنل نمی‌داند
core/watchdog.py   تعویض خودکار سرور خراب، بدون اطلاع پنل
core/panel.py      تنها فایلی که PasarGuard را می‌شناسد — فقط موقع bind اجرا می‌شود
app.py             API و رابط وب
pepectl.py         ابزار خط فرمان (کار روزمره از همین‌جا)
migrate.py         مهاجرت از نسخه قدیم
```

اگر `core/panel.py` را پاک کنید، هسته باز هم بی‌نقص کار می‌کند؛ فقط باید
پورت‌ها را دستی داخل Xray بنویسید.

مسیرهای API هم عمداً همین جدایی را نشان می‌دهند:

| مسیر | با پنل تماس می‌گیرد؟ |
|---|---|
| `/api/engine/*` | **خیر** — کلید، سرور، اسلات، سلامت |
| `/api/panel/*` | بله — فقط برای bind اولیه |

اگر بعد از bind اولیه هیچ‌وقت چیزی زیر `/api/panel/` صدا نزدید، یعنی سیستم
دقیقاً همان‌طور که طراحی شده کار می‌کند.

## نصب

```bash
sudo bash install.sh
```

بعد از نصب، `http://<server-ip>:8088` را باز کنید. تب **Setup** یک راهنمای
چهارمرحله‌ای است که تیک می‌خورد و کل کار از همان‌جا انجام می‌شود:

```
۱. افزودن کلید سرف‌شارک
۲. رزرو اسلات‌ها
۳. پر کردن خودکار با کشورها (هر اسلات یک کشور، پین‌شده)
۴. Bind به پنل  ← تنها مرحله‌ای که با پنل حرف می‌زند
```

هر کاری که از CLI برمی‌آید از پنل وب هم برمی‌آید: rotate کلید، پین و
جابه‌جایی کشور، ری‌استارت یا خالی کردن یک اسلات، پخش کلیدها، نمایش و
هم‌ترازسازی پورت‌ها، فعال/غیرفعال کردن هاست‌ها و unbind کامل.
CLI برای وقتی است که SSH دارید و مرورگر نه.

### راه‌اندازی اولیه

```bash
pepectl key add --key <SURFSHARK_PRIVATE_KEY>
pepectl slots ensure 15
pepectl slots fill

# این یکی فقط یک بار — تنها دستوری که با پنل حرف می‌زند
pepectl bind --host https://panel.tld --token <TOKEN> \
             --core-id <ID> --slot-count 15
```

اگر `--template-host-id` را ندهید، لیست هاست‌های قابل استفاده را نشان می‌دهد.

## قفل کشور روی پورت

هر اسلات به یک کشور **پین** می‌شود. یعنی:

```
:20004 → آلمان     (برای همیشه)
:20005 → ژاپن      (برای همیشه)
```

بعد از پین شدن، این‌ها **نمی‌توانند** کشور را عوض کنند:

- تعویض پرایوت کی — اصلاً به `endpoint` دست نمی‌زند
- سوختن سرور و جایگزینی واچ‌داگ — فقط داخل همان کشور می‌گردد
- `assign_server` با کشور اشتباه — خطای `CountryLockError` می‌دهد و رد می‌کند
- ریبوت سرور — از رجیستری همان کشور را بالا می‌آورد

```bash
# پین کردن
pepectl slot pin 4 --country Germany
pepectl slot pin 5 --country Japan

# دستور fill خودش موقع پر کردن پین می‌کند
pepectl slots fill
```

در خروجی `pepectl status` علامت 🔒 یعنی آن پورت پین شده.

### اگر کل یک کشور بخوابد

اگر هیچ سروری از کشور پین‌شده در دسترس نباشد، واچ‌داگ اسلات را مرده رها
نمی‌کند. ترتیب انتخابش این است:

```
۱. همان کشور        ← حالت عادی، کشور خروجی عوض نمی‌شود
۲. کشور همسایه      ← فقط وقتی کل کشور خوابیده
۳. هر سرور سالم     ← آخرین چاره
```

مهم این است که **پین پاک نمی‌شود**. اسلات فقط موقتاً جای دیگری است و به
محض اینکه کشور خودش برگردد، واچ‌داگ در سوییپ بعدی خودکار برش می‌گرداند
سر جایش. در `pepectl status` با ⚠ نشان داده می‌شود و در پنل وب هم برچسب
`temporarily via …` می‌خورد.

همسایه‌ها در `core/engine.py` داخل دیکشنری `NEIGHBOURS` تعریف شده‌اند و
می‌توانید تغییرشان دهید.

### اگر عمداً خواستی کشور یک اسلات را عوض کنی

```bash
pepectl slot relocate 4 --country Netherlands
```

عمداً دستور جداگانه است و تأیید می‌خواهد، چون کاربرهایی که روی اینباند
`PS-In-004` هستند بدون هیچ تغییری در سمت خودشان، از کشور جدید خارج می‌شوند.
این اتفاق نباید تصادفی بیفتد.

## کار روزمره

همه‌ی این‌ها **بدون پنل** انجام می‌شوند:

```bash
# تعویض پرایوت کی — دلیل اصلی این بازنویسی
pepectl key rotate <identity-id> --key <NEW_KEY>

# وضعیت
pepectl status
pepectl status --deep     # تست واقعی خروجی هر تونل

# ری‌استارت همه تونل‌ها روی همان پورت‌ها
pepectl recover
```

## اگر لوکیشن‌ها تصادفی قطع می‌شوند

اگر هر یکی دو ساعت بعضی لوکیشن‌ها می‌میرند و **فقط با تعویض پرایوت کی**
درست می‌شوند، علتش تقریباً همیشه این است: یک جفت‌کلید همزمان به سرورهای
زیادی وصل است. سرف‌شارک آن را یک هویت می‌بیند که از N جای مختلف وصل شده و
سشن‌های اضافه را قطع می‌کند. اینکه کدام‌ها قطع شوند تصادفی به نظر می‌رسد.

```bash
pepectl key load
```

اگر `OVERLOADED` دیدید، همین است.

### راه‌حل: کلید بیشتر، نه اکانت بیشتر

داخل بخش WireGuard اکانت سرف‌شارکتان می‌توانید **چند جفت‌کلید** بسازید.
نیازی به اشتراک اضافه نیست.

```bash
pepectl key add --key <KEY_2> --label key2
pepectl key add --key <KEY_3> --label key3
pepectl key add --key <KEY_4> --label key4
pepectl key rebalance
pepectl key load          # باید healthy شود
```

سقف پیشنهادی حدود ۵ تونل روی هر کلید است. با
`PEPECORE_MAX_SLOTS_PER_KEY` قابل تغییر است.

### چرخش زمان‌بندی‌شده

علاوه بر پخش بار، می‌توانید هر چند ساعت جای کلیدها را بین اسلات‌ها عوض
کنید تا هیچ کلیدی مجموعه‌ی سشن ثابت و طولانی‌مدت نداشته باشد:

```bash
pepectl rotate set --hours 3      # هر ۳ ساعت
pepectl rotate status
pepectl rotate now                # همین الان
pepectl rotate set --off
```

از پنل وب هم در تب **Keys** همین تنظیم هست. پورت‌ها، کشورها و کانفیگ
کاربرها دست‌نخورده می‌مانند و پنل اصلاً درگیر نمی‌شود. زمان‌بندی در
رجیستری ذخیره می‌شود، پس بعد از ریبوت هم ادامه پیدا می‌کند.

**چیزی که این نمی‌کند:** خودش وارد اکانت سرف‌شارک نمی‌شود و کلید جدید
نمی‌سازد. برای آن باید یوزر و پسورد اکانتتان روی سرور ذخیره شود و
اندپوینت‌های غیررسمی صدا زده شوند؛ ضمن اینکه ثبت و حذف خودکار کلید خودش
یکی از الگوهایی است که باعث بلاک شدن اکانت می‌شود. کلیدها یک بار دستی
اضافه می‌شوند.

### چند کلید همزمان

می‌توانید چند اکانت سرف‌شارک اضافه کنید تا بار بین آن‌ها پخش شود:

```bash
pepectl key add --key <KEY_2> --label backup
pepectl key rebalance
```

اگر یک اکانت سوخت، فقط همان را rotate کنید؛ اسلات‌های وابسته به کلیدهای دیگر
حتی ری‌استارت هم نمی‌شوند.

## ثابت بودن پورت‌ها بین سرورها

کل نگاشت پورت از `core/config.py` می‌آید، نه از فایل رجیستری:

```python
SOCKS_BASE_PORT   = 20000     # slot N → 20000+N
INBOUND_BASE_PORT = 10000     # slot N → 10000+N
```

یعنی اسلات ۴ روی **هر** سروری که این پروژه را اجرا کند `:20004` است — چه
همین الان، چه شش ماه بعد روی یک سرور تازه.

رجیستری هم موقع هر بار خوانده شدن با همین فایل هم‌تراز می‌شود. اگر
`registry.json` را از سرور دیگری کپی کنید که پورت‌هایش فرق داشته، خودش را
خودکار اصلاح می‌کند و پین کشورها هم سر جایشان می‌مانند.

```bash
pepectl ports show      # نمایش نگاشت کامل
pepectl ports resync    # هم‌ترازسازی اجباری با config.py
```

### اگر دو نسخه روی یک سرور می‌خواهید

```bash
PEPECORE_SOCKS_BASE=21000 PEPECORE_INBOUND_BASE=11000 pepectl status
```

`config.py` موقع لود چک می‌کند که دو رنج با هم تداخل نداشته باشند و اگر
داشته باشند همان اول خطا می‌دهد.

**هشدار:** اگر روی سیستم زنده این عددها را عوض کنید، کانفیگ کوری که در
پنل دارید به پورت‌های اشتباه اشاره می‌کند. بعدش حتماً `ports resync` و
سپس یک بار `bind` بزنید.

## مهاجرت از نسخه قدیم

```bash
python3 migrate.py --from /root/pepepanel/pepeshark --dry-run   # پیش‌نمایش
python3 migrate.py --from /root/pepepanel/pepeshark
systemctl restart pepecore
pepectl bind --host … --token … --core-id … --slot-count <N>
```

نصب قدیمی دست‌نخورده باقی می‌ماند؛ چیزی پاک نمی‌شود.

**یک نکته مهم:** این آخرین باری است که کاربرها باید لینک سابسکریپشن جدید
بگیرند. تگ‌های قدیمی (`B-In-*`) با شبکه‌ی ثابت `PS-In-*` جایگزین می‌شوند.
بعد از این یک بار، دیگر هیچ‌وقت لازم نیست re-inject بزنید.

## واچ‌داگ

هر ۲ دقیقه همه تونل‌ها را چک می‌کند. اگر یکی دو بار پشت‌سرهم جواب نداد،
خودکار یک سرور دیگر از همان کشور جایگزین می‌کند و تونل را بالا می‌آورد.

پورت عوض نمی‌شود، تگ عوض نمی‌شود، پس Xray همچنان درست روت می‌کند و کاربر
اصلاً متوجه نمی‌شود. در معماری قبلی همین سناریو نیاز به دخالت دستی و
re-inject داشت.

## نکات فنی

- **پورت مبدأ UDP**: `ListenPort` عمداً در کانفیگ نوشته نمی‌شود تا کرنل به هر
  تونل یک پورت ephemeral تصادفی بدهد. پورت‌های ثابت و پشت‌سرهم باعث می‌شوند
  conntrack جواب‌ها را بین تونل‌ها قاطی کند، که خودش را به شکل خطای
  `invalid mac1` نشان می‌دهد.
- **دسترسی فایل‌ها**: `data/wg/` با مجوز `700` و کانفیگ‌ها با `600` ساخته
  می‌شوند چون پرایوت کی داخلشان است.
- **نوشتن رجیستری**: اتمیک است (نوشتن روی فایل موقت و بعد `os.replace`)، پس
  قطع برق وسط کار فایل نیمه‌نوشته باقی نمی‌گذارد.
- **رشد اسلات‌ها**: `ensure_slots` هیچ‌وقت اسلات موجود را حذف یا شماره‌گذاری
  مجدد نمی‌کند. فقط اگر تعداد را زیاد کنید لازم است دوباره bind بزنید، و
  اسلات‌های قبلی هم UUID خودشان را نگه می‌دارند.
